# 赫朝服务器登录器与分发平台实施文档

> 文档状态：`0.7.1` 启动器、API 源码 `0.8.0`（线上 `0.6.0`）、状态采集器 `0.1.0` 与 Velocity 授权灰度基线
> 更新日期：2026-07-24
> 范围：玩家登录器、客户端分发、账号与权限、管理员后台、阿里云后端、Minecraft VPS 接入、HTTPS 与迁移

## 1. 项目目标

把当前启动器升级为整个赫朝 Minecraft 服务器的统一登录入口，而不只是活动服启动器。

玩家使用一个 Windows 客户端完成：

- 登录并绑定正版 Minecraft 身份。
- 查看自己有权限进入的大厅、生存服和活动服。
- 自动安装或更新原版、Paper 兼容客户端、Forge、NeoForge、Fabric 等不同客户端档案。
- 校验客户端文件，缺失时修复，更新失败时安全回滚。
- 从启动器直接进入统一 Velocity 入口，再由代理转入目标服务器。
- 查看服务器状态、在线人数、维护公告和更新进度。

管理员通过独立 Web 控制台完成：

- 管理服务器目录、活动、客户端档案、版本和公告。
- 管理玩家等级、白名单和服务器进入权限。
- 发布、暂停、回滚客户端版本。
- 查看下载、登录、权限修改和发布审计记录。
- 查看游戏服务器状态。第一阶段只读，不让玩家端或平台自动启动 Minecraft 服务端。

## 2. 已确认原则

- 游戏 VPS 负责 Velocity 和 Minecraft 服务端，不承担大规模客户端文件分发。
- 阿里云负责 HTTPS API、账号、权限、清单和管理后台。
- 大文件优先放阿里云 OSS，并通过 CDN 或 OSS 下载域名分发。
- `hechao.world` 现有网站必须保留。
- `api.hechao.world` 已用于中转站，不能直接改成启动器 API。
- 启动器 API 使用新子域名，避免影响现有业务。
- 玩家能否进服不能只由客户端判断，客户端只负责展示，Velocity 或后端必须再次强制校验。
- 客户端不保存服务器 SSH 密码、阿里云密钥、管理密钥或数据库密码。
- 所有 Minecraft 服务端默认由管理员手动启动。平台建设期间不得自动启动任何服。
- 新版本先发布到测试通道，小范围验证后再切换正式通道。
- 官网、启动器及发布材料持续展示非官方产品声明，不暗示获得 Mojang 或 Microsoft 认可。
- 只分发自有模组、配置与资源，不重新分发 Minecraft 本体或经修改的官方游戏文件。

## 4. 推荐总体架构

```text
玩家启动器
  | HTTPS 登录、目录、权限、清单
  v
阿里云 API / 管理后台 / 数据库
  |
  +---- OSS + CDN：JRE、模组、配置、资源包、启动器更新
  |
  +---- 状态采集：只读取游戏 VPS 状态

玩家 Minecraft 客户端
  | Minecraft 协议
  v
游戏 VPS 的 Velocity 统一入口
  |
  +---- 大厅
  +---- survival2 生存服
  +---- NeoForge / Forge / Fabric 活动服
  +---- DollNight 等插件活动服
```

### 4.1 游戏 VPS

职责：

- 运行 Velocity、大厅、生存服和活动服。
- 向状态采集服务提供在线状态、人数、版本和维护状态。
- 在 Velocity 或后端服执行最终进服权限判断。
- 保存世界、插件、模组和运行日志。

不承担：

- 玩家客户端下载。
- 用户密码登录。
- 公网管理后台。
- 将 SSH 能力直接暴露给启动器。

### 4.2 阿里云

职责：

- 登录、令牌、账号绑定和权限 API。
- 服务器目录和客户端档案 API。
- 管理员 Web 控制台。
- 发布清单、审计日志和状态缓存。
- OSS/CDN 文件分发。
- HTTPS 证书终止和反向代理。

阿里云 ECS 适合运行轻量 API，客户端下载流量应尽量交给 OSS/CDN，避免带宽和系统盘成为瓶颈。

## 5. 域名规划

| 域名 | 用途 | 处理原则 |
| --- | --- | --- |
| `hechao.world` | 现有网站 | 保持现状，不在首轮迁移中改业务 |
| `api.hechao.world` | 现有中转站 API | 保持接口和监听，迁移时只做兼容代理 |
| `launcher-api.hechao.world` | 启动器与管理后台 API | 已新增，A 记录指向阿里云主机并通过 HTTPS 代理到回环 API |
| `download.hechao.world` | OSS/CDN 客户端文件 | 新增，CNAME 到 OSS/CDN |
| `admin.hechao.world` | 管理员控制台 | 已新增 HTTPS 入口；生产仍返回 404，源码控制台已完成 Host 锁定与 MFA，待部署灰度 |

可以使用一张包含上述子域名的 SAN 证书，或使用 DNS 验证签发通配符证书。证书私钥只能保存在阿里云反向代理上。

## 6. HTTPS 与迁移策略

配置 HTTPS 本身不会必然导致 `api.hechao.world` 或 `hechao.world` 无法访问。真正的风险来自端口冲突、错误反向代理、DNS 提前切换和原服务监听地址被直接修改。

### 6.1 迁移前盘点

- [x] 记录阿里云实例的 CPU、内存、系统盘、数据盘、公网带宽和操作系统（带宽计费规格仍需控制台复核）。
- [x] 记录当前 `80`、`443` 及业务端口的监听进程。
- [x] 导出当前 Nginx、Caddy、IIS 或应用自身的监听配置。
- [x] 记录 `hechao.world` 和 `api.hechao.world` 的 DNS 解析、TTL 和证书状态。
- [x] 逐项记录现有网站和中转 API 的健康检查 URL。
- [ ] 创建系统快照，并单独备份配置、数据库和上传文件。
- [ ] 确认回滚负责人、回滚命令和最长可接受中断时间。

实时盘点见 [`ASSET_INVENTORY.md`](ASSET_INVENTORY.md)。当前已有一次可校验的迁移归档，但尚未复核云快照、每日备份和恢复演练，因此最后两项保持未完成。

### 6.2 安全上线顺序

1. 先部署新的启动器 API，监听 `127.0.0.1` 的独立高位端口。
2. 不修改现有网站和中转 API，先用本机请求验证新 API。
3. 使用当前已有的反向代理增加新子域名，不能再安装一个争抢 `80/443` 的代理。
4. 若当前业务直接占用 `80`，先用 DNS 验证签发证书，再安排一次有回滚方案的监听切换。
5. 新增 `launcher-api.hechao.world`，只把新域名指向新 API。
6. 新增 `download.hechao.world`，指向 OSS/CDN。
7. 分别验证旧网站、旧中转 API、新 API 和下载域名。
8. 观察错误率和延迟后，再考虑是否给现有域名补齐 HTTPS。

截至 2026-07-23，步骤 1 至 7 已完成：API `0.6.0` 仅监听 `127.0.0.1:8090`，`launcher-api.hechao.world` 已通过独立 Nginx server block 和 ZeroSSL 证书提供 HTTPS，`admin.hechao.world` 已预留有效 HTTPS 入口，`download.hechao.world` 已连接私有 OSS 并启用 HTTPS。数据库、真实目录、认证会话、LuckPerms 同步、签名分发、Velocity 授权和只读状态心跳均已接入，旧网站与旧中转 API 回归检查仍为 HTTPS 200；步骤 8 持续观察仍在进行。

### 6.3 系统迁移原则

- [ ] 不在原机直接重装系统。
- [ ] 如需换系统，创建新实例或新系统盘，完成平行部署和验证后再切 DNS。
- [ ] DNS 切换前至少提前一天降低 TTL。
- [ ] 新旧环境至少并行保留一个观察周期。
- [ ] 数据库切换前停止写入或使用可验证的增量同步。
- [ ] 旧环境在确认无误前不删除。

### 7.2 等级与服务器权限

等级可以决定默认权限，但数据库应保存明确授权，避免等级规则变化后难以处理例外。

当前 LuckPerms 等级由社区管理员依据参与和运营职责分配，不通过付费、站外购买或外部产品获得；任何等级都不能绕过 Minecraft 正版权益校验。

```text
用户 -> 角色 -> 权限
用户 -> 服务器例外授权或拒绝
服务器 -> 最低等级 + 开放时间 + 白名单策略
```

- [x] 定义现有称号到四级 `AccessTier` 的映射。
- [x] 确定 LuckPerms 共享 MariaDB 为称号权威来源。

- [x] Velocity 插件在连接时异步校验 UUID 和服务器授权；已安装为 `monitor`，待真实账号灰度后切换 `enforce`。
- [x] 权限同步缓存时间为 5 分钟；登录后目录请求读取最新已同步结果。
- [ ] 所有管理员授权、拒绝和等级修改写入审计日志。

客户端自动识别只用于改善界面，不能作为唯一防线。修改本地文件或伪造 API 响应不能获得进服权限。

## 8. 服务器目录与客户端档案

服务器目录至少需要以下字段：

- 服务器 ID、显示名、图标和排序。
- 状态、在线人数、最大人数和维护公告。
- Minecraft 版本、加载器类型和加载器版本。
- Velocity 目标服务器 ID。
- 最低等级、白名单和开放时间。
- 对应客户端档案 ID。
- 是否在玩家端显示、是否允许进入。

客户端档案至少需要以下字段：

- 档案 ID、显示名、版本和发布通道。
- Minecraft、Java、加载器版本。
- 主清单 URL、清单签名和发布时间。
- JVM 参数、推荐内存和最大内存。
- 启动前检查规则和目标 Velocity 地址。
- 是否强制更新、最低兼容版本和回滚版本。

同一个基础文件只存一份。大厅和 `survival2` 可共用基础档案；活动服通过继承基础档案叠加自己的模组、配置和资源包。

## 9. 客户端下载与安装

### 9.1 发布格式

每个客户端版本发布一份签名 JSON 清单。每个文件记录：

```json
{
  "path": "mods/example.jar",
  "size": 3238514,
  "sha256": "...",
  "url": "https://download.hechao.world/objects/...",
  "required": true
}
```

清单还应包含档案版本、Minecraft 版本、Java 版本、加载器版本、删除列表和数字签名。

### 9.2 启动器更新流程

1. 获取当前用户可用服务器和客户端档案。
2. 下载并验证签名清单。
3. 对比本地文件的路径、大小和 SHA-256。
4. 将缺失或变化的文件下载到临时目录。
5. 支持 HTTP Range 断点续传、超时、重试和限并发。
6. 下载结束后逐文件校验 SHA-256。
7. 使用目录重命名完成原子切换。
8. 保留上一个完整版本用于回滚。
9. 启动成功后再清理过旧缓存。

### 9.3 必须处理的情况

- [x] 首次安装和全文件级增量更新。
- [x] 用户手动删改模组后的修复。
- [x] 磁盘空间不足。
- [x] 下载中断、代理错误和文件损坏。
- [ ] OSS/CDN 暂时不可用。
- [x] 服务端更新后客户端仍是旧版本。
- [x] 不同活动服使用同名但不同版本模组。
- [x] 强制删除已下线模组和旧配置。
- [x] 杀毒软件占用文件导致无法替换并恢复当前版本。
- [ ] 客户端版本回滚。

第一版使用全文件级增量更新即可，不必先做二进制差分。文件去重、压缩和 CDN 缓存已经能解决大部分带宽问题。

## 14. 容量评估

现阶段不能仅凭“VPS”和“阿里云”判断配置是否足够，必须先记录实际规格和峰值数据。

### 14.1 需要采集

- [ ] 游戏 VPS 的 CPU 型号、核心数、内存、磁盘类型和剩余空间。
- [ ] 各 Minecraft 进程的 `Xms/Xmx`、峰值 RSS、CPU 和磁盘写入。
- [ ] Velocity、大厅、生存服和活动服同时运行时的余量。
- [ ] 20 至 30 人活动时的 TPS、MSPT、GC 暂停和网络出站。
- [ ] 阿里云 ECS 的 CPU、内存、系统盘和公网带宽。
- [ ] 当前中转 API 的峰值 QPS、响应时间和流量。
- [ ] 单个客户端完整包大小、日更新人数和预计并发下载数。

### 14.2 判断标准

- 游戏开始后 MSPT 持续低于 50 ms，避免 TPS 长时间低于 20。
- Minecraft 满载时系统仍保留内存和 CPU 余量，不能依赖交换分区维持。
- API 峰值 CPU、内存、连接数和数据库连接池有至少约 30% 余量。
- 客户端下载不会经过游戏 VPS。
- 阿里云 API 带宽不承载完整模组包，文件由 OSS/CDN 直接下发。
